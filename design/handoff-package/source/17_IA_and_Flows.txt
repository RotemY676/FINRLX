This document defines the canonical route map, workspace hierarchy, and user flows for QuantPipeline. It is the connective tissue between product scope, UX principles, and the implementation contract used by frontend and backend teams.

# 1. Information architecture principles
Organize around decisions, not around model vendors or data-source silos.
Treat validation and operations as first-class routes, not buried utilities.
Allow the user to move from summary to evidence to governance without breaking context.
Preserve a stable shell so route changes feel like focus changes, not product changes.

# 5. Primary user flows
The flows below are the minimum journeys that the design and QA process must explicitly validate.

## Daily review flow
Open overview
Inspect top recommendation card
Enter decision workspace
Read evidence and risk summary
Save, defer, or move to paper
Required outcome: A recorded decision state with timestamp and notes

## Challenge flow
Open a recommendation
Navigate to comparison
Inspect divergence and attribution
Change one or more assumptions
Return to decision workspace with delta preview
Required outcome: A revised recommendation or an explicitly deferred decision

## Replay investigation flow
Open replay from incident or paper alert
Load historical snapshot
Inspect inputs, state, and policy status
Compare proposed action vs actual result
Write ops note or create follow-up issue
Required outcome: A resolved investigation artifact

## Operator publication flow
Open ops command center
Inspect publication queue item
Check policy and health gates
Approve, hold, or rollback
Record explanation and actor
Required outcome: A controlled release action visible in audit history

# 7. Navigation rules and anti-patterns
Do not create one route per data source if the user expects one decision workspace.
Do not hide validation and replay behind admin-only patterns; they are part of core trust.
Do not require the user to reconstruct the current recommendation when moving between comparison and risk.
Avoid modal overload for critical flows; complex review belongs in full workspaces or drawers.

# 9. Flow instrumentation requirements
Every key flow should emit entry, exit, and decisive action events.
Instrumentation should distinguish passive viewing from active challenge behavior.
Abandonment points in challenge and publication flows need explicit event markers.
Route-level analytics should always include current persona, mode, and scope.
