This document defines the first component baseline for QuantPipeline. It is intentionally product-specific: components are described not only by appearance but by the analytical and operational jobs they must support.

# 1. Component system principles
Components should encode product meaning, not just generic layout.
Every component must define states, empty conditions, loading behavior, and error behavior.
Keyboard and accessibility requirements are part of the component contract.
Permissions and role differences should be handled through configuration, not ad-hoc duplication.

# 3. Detailed component contracts

## Recommendation card
Purpose: Presents one recommendation summary in overview or mobile contexts.
Required states: default, highlighted, provisional, saved, stale, unavailable
Must define a compact variant and a full-detail variant where relevant.
Must expose explicit status labels; color alone is insufficient.
Must specify loading, empty, and failure behavior.
Must define analytics events for decisive interactions.

## Confidence block
Purpose: Separates model confidence, data quality, and operational readiness.

Must expose explicit status labels; color alone is insufficient.
Must specify loading, empty, and failure behavior.
Must define analytics events for decisive interactions.

# 6. Density and responsiveness rules
Components should define responsive collapse behavior instead of relying on arbitrary CSS shrinkage.
Dense desktop variants may exist, but they must still preserve readable headings and action affordances.
Mobile variants should prioritize one primary insight and one primary action.

# 7. Tokenization guidance
Expose spacing, radius, border, and elevation tokens consistently.
Separate semantic color tokens from raw palette values.
Text roles should use a documented token map: page title, section title, body, meta, table label.

# 12. Component sign-off checklist
The component name, purpose, states, and accessibility rules are documented in one place.
A frontend implementer can tell which props are required and which are optional.
QA can derive visual, functional, and role-aware test cases from the spec without product guesswork.
